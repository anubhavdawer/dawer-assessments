Codebase Inheritance Assessment — calcom/cal.com (forked as "Cal.diy")

Executive Summary

This is a very large, mature, production-grade scheduling platform — a Turborepo monorepo containing 7,690+ files across a Next.js 16 web app, a NestJS "API v2" platform, a tRPC backend, a large Prisma/PostgreSQL data layer, an extensive app-store integration system, and React embed packages. The repository is the open-source Cal.com codebase rebranded as a community fork ("Cal.diy"); however, the actual code still contains a large amount of commercial/"enterprise" surface area (Stripe billing, organizations, SAML/SCIM, Cal.ai voice agents, credit billing), so the README's claim that "enterprise code has been removed" is not reflected in the schema, env vars, or modules we read. This is healthy, actively maintained software with strong CI, but it is enormous, deeply coupled, and operationally demanding: it requires Postgres, Redis, an SMTP server, dozens of third-party OAuth credentials, and careful secret management. Inheriting it safely is an XL undertaking — onboarding alone is multi-week, and "full ownership" implies a team, not an individual. The biggest risks are sheer surface area/complexity, a sprawling 100KB+ Prisma schema with no clear soft-delete strategy, an env-var matrix in the hundreds, and the gap between the "stripped-down fork" narrative and the still-commercial reality of the code.

1. Project Summary

  • Purpose: Self-hostable scheduling/booking platform (Calendly competitor). Users create event types, share booking pages, sync calendars (Google/Office365/CalDAV), run video calls (Daily.co), accept payments (Stripe), and automate workflows (SMS/email/WhatsApp).
  • Current state: Highly mature, actively developed. Web app is at version 6.2.0; uses bleeding-edge dependencies (Next.js 16.2.3, React 18 with React 19 resolutions present, Turbo 2.7, Yarn 4.12, Biome 2.3).
  • Maturity: 🟢 Production-deployed at scale by the upstream vendor. This is not a greenfield or prototype codebase. The risk is not immaturity — it's complexity and the fork's divergence from upstream.
  • Fork caveat: 🟡 README describes "Cal.diy" as MIT-only with EE features removed. The schema and env still contain Organizations, SAML, Stripe billing, credits, Cal.ai, delegation credentials, etc. Either the removal is incomplete or the README is aspirational. This narrative/reality gap must be clarified before scoping.

3. Architecture

A Turborepo monorepo with clear workspace boundaries:

  • apps/web — Next.js front-end + legacy API routes (/api/*). Hybrid App Router + Pages Router.
  • apps/api/v2 — Standalone NestJS "Platform API" with its own module system, throttling, Bull queues, Swagger. Deployable to Vercel serverless (singleton Express handler in main.ts) or as a container.
  • packages/trpc — The primary internal API consumed by the web app. Root router → viewerRouter with ~30 sub-routers (bookings, eventTypes, slots, availability, admin, apiKeys, webhooks, oAuth, etc.).
  • packages/features — Domain logic (auth, tasker, webhooks, CRM). This is where the real business logic lives — repository + service patterns are used (UserRepository, CredentialRepository, ProfileRepository).
  • packages/prisma — Schema, migrations, Kysely + Zod + enum codegen.
  • packages/app-store — Plugin architecture for ~70+ integrations, with a CLI to scaffold apps and generated index files.
  • packages/embeds, packages/platform, packages/ui — Embeds, public SDK, design system.

Patterns: Repository/service layering in features, codegen-heavy (Prisma → Zod/Kysely/enums; app-store CLI generates *.generated.ts). 🟡 There are two parallel API stacks (tRPC for the web app, NestJS REST for the platform) plus legacy Next.js API routes — three ways to reach the backend, all sharing the same Prisma layer. This is powerful but a significant cognitive and maintenance burden.

4. Dependency Health

  • 🟢 Dependencies are largely current and aggressively maintained — Next 16, Tailwind 4, Yarn 4, Biome. Far healthier than typical inherited codebases.
  • 🟡 Large resolutions block (~80 entries) in root package.json pinning transitive deps (axios, ajv, qs, jsonwebtoken, multer, tar, lodash, etc.) — this is a manual CVE-mitigation treadmill. It works but is fragile and signals frequent supply-chain firefighting. Whoever inherits this owns that treadmill.
  • 🟡 Version skew across workspaces: Stripe 9 (web) vs 15 (api); Sentry 10 vs 9; Prisma client 5.4.2 (patched) vs internals 6.16; Turbo 2.7 vs 2.5; @microsoft/microsoft-graph-types-beta 0.15 vs 0.42. Drift like this causes subtle bugs and complicates upgrades.
  • 🟡 Patched packages in .yarn/patches: Prisma client (29KB patch!), dayjs, next-i18next, libphonenumber. A 29KB Prisma patch is a serious maintenance liability — every Prisma upgrade risks breaking it.
  • 🟡 react pinned at 18.2.0 while react@19.2.x resolutions exist — the codebase is mid-React-19-migration, an unstable state to inherit.
  • 🟢 EOL runtime check: Node 20 (Dockerfile) is in maintenance LTS, acceptable through 2026. README claims Node >=18 (18 is EOL April 2025 🟡 — should be bumped to 20+).
  • 🟢 A security-audit.yml workflow runs yarn npm audit and fails on critical (but not high) severity.

5. Infra & Deployment

  • 🟢 Extensive CI/CD: ~60 GitHub Actions workflows — typed checks, lint, unit, integration, multiple E2E suites (web, embed, app-store, api-v2, atoms), bundle-analysis, prisma-migration checks, changesets, docker release, and ~10 cron workflows (cron-*.yml) acting as the scheduler.
  • 🟢 Multi-stage Dockerfile (builder → placeholder-replace → runner) with turbo prune, healthcheck, and a runtime start.sh that does find/replace of the build-time NEXT_PUBLIC_WEBAPP_URL placeholder. Sophisticated.
  • 🟢 docker-compose.yml with Postgres, Redis, web, api v2, and optional Prisma Studio. One-command yarn dx local bootstrap.
  • 🔴 Build requires a live database (README explicitly states this). This is a real CI/CD friction point and an anti-pattern — builds shouldn't depend on a running DB.
  • 🟡 Cron-as-CI: Scheduled jobs run as GitHub Actions cron workflows hitting authenticated endpoints with CRON_API_KEY. Portable but couples your scheduler to GitHub and is invisible if you migrate off it. Self-hosters must replicate these crons themselves.
  • 🟡 Deployment targets documented for Vercel (Pro required), Railway, Render, Northflank, Elestio. Vercel-centric assumptions (@vercel/functions, @vercel/og, waitUntil, edge config) are baked into auth code.

6. Code Health

  • 🟢 Strong typing discipline — TS strict, codegen for DB types, tRPC end-to-end types, Zod everywhere. Biome enforces lint/format with lint-staged + husky pre-commit.
  • 🟢 Repository/service abstractions in features keep DB access reasonably organized.
  • 🟡 Massive files / hot-spots: next-auth-options.ts is 45KB of branching identity-merge logic — one of the most complex and security-sensitive files in the repo, with deeply nested conditionals for account linking/merging. The Prisma schema is 101KB. These are change-averse, high-blast-radius files.
  • 🟡 Numerous // deprecated … plan to drop this column markers (EventType.price, currency, weightAdjustment, Booking.scheduledJobs, User.organizationId). Dead/legacy fields accumulate; migration debt is acknowledged but unpaid.
  • 🟡 console.error/console.warn interspersed with the structured logger in auth code — inconsistent logging.
  • 🟢 Test infrastructure is serious (Vitest + Playwright + Jest, prismock for DB mocking, multiple E2E projects). Coverage is uneven across the huge surface but the harness is mature.

7. Completeness

  • 🟢 Core product is fully implemented and battle-tested: event types, booking flow, availability/slots engine, calendar sync, video, payments, workflows, embeds, public API v2 with Swagger.
  • 🟡 The fork's stated scope is incomplete/contradictory. README says Teams/Orgs/Insights/Workflows/SAML/SSO removed; the schema (Profile, Team, OrganizationSettings, DSyncData), env vars (SAML_, ORGANIZATIONS_, Stripe org pricing), and NestJS modules still fully contain them. Treat "removed" claims as unverified until audited.
  • 🟡 endpoints.module.ts has // TODO: apply ratelimits on the EndpointsModule configure block — rate-limiting is handled by the global guard but the module-level TODO suggests intended granular limits never landed.
  • 🟡 Optional/half-wired features gated behind env flags: Trigger.dev async tasker (ENABLE_ASYNC_TASKER=false), Unkey rate limiting (optional), Cal.ai voice (RETELL_AI_*), credits billing.

8. Landmines

  • 🔴 Default/placeholder secrets ship in config. .env.example ships a real-looking CRON_API_KEY='0cc0e6c35519bba620c9360cfe3e68d0'; Dockerfile defaults NEXTAUTH_SECRET=secret and CALENDSO_ENCRYPTION_KEY=secret; docker-compose Postgres uses unicorn_user/magical_password. The README warns about these, but the defaults are footguns that will reach production if anyone is careless.
  • 🔴 NODE_TLS_REJECT_UNAUTHORIZED=0 is recommended in the README for load-balancer SSL termination, and a dev script uses it. This globally disables TLS verification — a dangerous documented workaround.
  • 🟡 allowDangerousEmailAccountLinking: true on both Google and Azure AD providers. The code adds compensating checks (email-verified / xms_edov), but this remains an account-takeover-sensitive area; the auth file even contains explicit "Prevent account pre-hijacking" comments showing this has been a real concern.
  • 🟡 Identity-merge complexity in next-auth-options.ts — multiple branches that mutate user.email/identityProvider on login. High-risk surface for auth bugs; any change needs security review.
  • 🟡 Encryption key handling: backup codes and 2FA secrets are symmetric-encrypted with CALENDSO_ENCRYPTION_KEY read directly from process.env at call sites; key rotation is not obviously supported.
  • 🟡 Schema breadth invites N+1 and slow-query risk in the slots/availability engine (the most performance-critical path); env vars like NEXT_PUBLIC_QUERY_* and SLOTS_CACHE_TTL exist precisely to tune around this.
  • 🟡 bodyParser: false + manual raw/json/urlencoded middleware wiring in NestJS — correct for Stripe webhooks but fragile if routes are added without updating app.module.ts.

9. Production Readiness Gaps

  1. 🔴 Rotate/replace every default secret (NEXTAUTH_SECRET, CALENDSO_ENCRYPTION_KEY, CRON_API_KEY, DB creds). Verify none of the shipped placeholders are live in the target deployment.
  2. 🔴 Remove the live-looking CRON_API_KEY value from .env.example (it should be blank).
  3. 🔴 Eliminate the DB-required-at-build-time constraint or provision a throwaway DB in the build pipeline.
  4. 🟡 Provision and operate Redis (rate limiting, Bull queues, throttler storage) and SMTP — both are hard runtime dependencies.
  5. 🟡 Stand up all cron jobs outside GitHub if not deploying via the provided Actions (reminders, webhooks, timezone changes, downgrades, etc.).
  6. 🟡 Configure Sentry/Axiom/PostHog or accept blind operation; bump Node to 20+ in README.
  7. 🟡 Audit the fork: confirm what is actually disabled vs. what merely appears removed (orgs, SAML, billing).
  8. 🟡 Reconcile dependency version skew (Stripe, Sentry, Prisma) before first major upgrade.

Fullstack-Specific Sections

11. Data Model

  • Scale: ~100KB Prisma schema, dozens of models. Core: User, Team, Profile, Membership, EventType, Booking, Attendee, Credential, Schedule, Availability, BookingReference, plus billing (CreditBalance, *Log, MonthlyProration), orgs (OrganizationSettings, DSyncData), and AI (Agent, CalAiPhoneNumber).
  • 🟢 Indexing is thoughtful — Booking has 12+ targeted composite indexes (e.g. [startTime, endTime, status], [userId, status, startTime]), showing real query-pattern awareness.
  • 🟢 Sensible separation: UserPassword split from User to avoid leaking hashes.
  • 🟡 No consistent soft-delete pattern. Most models hard-delete via onDelete: Cascade; a few have ad-hoc deleted Boolean? (e.g. BookingReference). No tenant-wide soft-delete or audit-trail convention. Cascade deletes are wide-reaching — deleting a User cascades through credentials, event types, bookings, profiles.
  • 🟡 Acknowledged dead columns awaiting migration (EventType.price/currency, Booking.scheduledJobs, Host.weightAdjustment, User.organizationId). Migration debt.
  • 🟢 Migration state is managed (packages/prisma/migrations, CI check check-prisma-migrations.yml, db-deploy vs db-migrate).

12. API Surface

  • Three surfaces sharing one Prisma layer:
  • 🟢 API v2 has real REST hygiene: request-id middleware, response interceptors, throttling guard, raw-body handling for webhooks.
  • 🟡 Two public-facing API generations imply migration debt (v1 → v2); confirm v1 status in the fork.

13. Auth & Authorization

  • Authentication: NextAuth (JWT strategy) with a custom Prisma adapter. Providers: Credentials (email/password + TOTP/backup codes), Google, Azure AD, Email magic links, and SAML-IdP (via BoxyHQ @boxyhq/saml-jackson). API v2 uses passport-jwt + OAuth2 + API keys (API_KEY_PREFIX=cal_).
  • 🟢 2FA implemented (TOTP via otplib, encrypted backup codes). Rate limiting on login via checkRateLimitAndThrowError. Admin role-elevation requires valid password + 2FA in prod (degrades to INACTIVE_ADMIN otherwise).
  • Authorization: Role-based — UserPermissionRole (USER/ADMIN), MembershipRole (MEMBER/ADMIN/OWNER), plus a custom Role model (RBAC) referenced by Membership.customRoleId. Org-scoped via Profile.
  • 🔴/🟡 Known gaps: allowDangerousEmailAccountLinking: true (mitigated but sensitive); the 45KB identity-merge logic is the single highest-risk file; admin 2FA enforcement is bypassed in E2E and dev modes (NEXT_PUBLIC_IS_E2E, isENVDev) — ensure those flags are never truthy in prod.

14. Async Layer

  • Tasker (packages/features/tasker) — internal job abstraction with typed payloads (sendWebhook, sendSms, triggerHostNoShowWebhook, translateEventTypeData, createCRMEvent, bookingAudit, webhookDelivery…). Pluggable backends: "internal" | "redis".
  • Bull + Redis in API v2 for queues.
  • Trigger.dev integration, gated behind ENABLE_ASYNC_TASKER=false (off by default) — a parallel async system, partially adopted.
  • Cron via ~10 GitHub Actions workflows (reminders, webhooks, timezone changes, downgrades, digest emails, SMS/WhatsApp reminders).
  • 🟡 Multiple overlapping async mechanisms (Tasker/internal, Tasker/redis, Bull, Trigger.dev, GH cron) — understanding which path a given job takes is non-trivial.

15. Environment Variables

  • 🔴 Enormous surface: turbo.json globalEnv lists 300+ variables; .env.example is 17.6KB; a separate .env.appStore.example (4.1KB) exists. This is one of the largest env matrices we've assessed.
  • 🟢 .env.example and .env.appStore.example exist and are well-commented; dotenv-checker validates .env against the schema (env-check:common, env-check:app-store) as a dev/dx prerequisite.
  • 🔴 .env.example ships a concrete CRON_API_KEY; Dockerfile defaults secrets to secret. (Repeated from Landmines — it's that important.)
  • 🟡 No first-class secrets manager assumed; Infisical is referenced in scripts (infisical run) but optional. Self-hosters are on their own for secret storage.

16. State Management

  • 🟢 Front-end state is modern and restrained: Jotai for global atoms, TanStack Query (5.x) for server state via tRPC, nuqs for URL-synced state, react-hook-form + Zod for forms, next-themes for theming, kbar for command palette.
  • 🟢 No Redux/MobX bloat. Server state is the source of truth via tRPC/React Query — a healthy, conventional pattern. Complexity is in volume of features, not in state-management architecture.

17. Routing

  • Web: Next.js hybrid App Router + Pages Router (mid-migration). Booking pages are dynamic ([user], [type], org subdomains). Code splitting via Next + dynamic import() (seen throughout auth code for lazy modules like signJwt, totp).
  • API v2: NestJS controller routing with custom middleware ordering and RewriterMiddleware/RedirectsMiddleware.
  • 🟡 Dual-router state in apps/web increases the chance of inconsistent layouts/loading behavior and complicates onboarding. Org subdomain routing depends on ALLOWED_HOSTNAMES/RESERVED_SUBDOMAINS and external DNS (Vercel/Cloudflare), adding deployment complexity.

18. Design System

  • 🟢 Dedicated @calcom/ui package (plus @coss/ui) built on Radix UI primitives with Tailwind CSS 4. Consistent token-based theming (brandColor, darkBrandColor, theme, appTheme per user/team). next-themes for dark mode. Atoms are published as an npm package (@calcom/atoms) with its own build/E2E.
  • 🟢 Web-design-guidelines and Vercel React best-practice "skills" are checked into .opencode/.cursor — indicates intentional design/perf standards.
  • 🟡 Mixed component sources (@calcom/ui, @coss/ui, react-select, sonner, raw Radix, react-phone-input-2) — some redundancy/inconsistency across the long history.

19. Performance Flags

  • 🟢 Bundle budget enforced: nextBundleAnalysis budget 350KB with CI annotation workflows (nextjs-bundle-analysis*.yml). Dedicated performance-tests.yml.
  • 🟢 SSR/SSG mix, sharp for images, @vercel/og for OG images, Turbopack dev.
  • 🟡 Slots/availability is the performance hot-spot — multiple tuning env vars (NEXT_PUBLIC_QUERY_AVAILABLE_SLOTS_INTERVAL_SECONDS, SLOTS_CACHE_TTL, SLOTS_WORKER_POOL_SIZE, ENABLE_SLOTS_WORKERS, NEXT_PUBLIC_QUICK_AVAILABILITY_ROLLOUT) reveal this is computationally heavy and needs Redis caching + worker pools to scale.
  • 🟡 React 18→19 migration in flight could affect rendering/CWV until stabilized. Memory pressure is real — README and scripts repeatedly bump --max-old-space-size to 6–16GB for builds.

20. Coupling Assessment

  • 🟡 Front-end and backend are tightly coupled in apps/web: the Next.js app, tRPC backend, and legacy /api routes all live together and share the @calcom/prisma, @calcom/features, and @calcom/lib workspaces directly (server code imported into the app at module level). They cannot be deployed fully independently.
  • 🟢 The *NestJS API v2 is independently deployable* (separate Dockerfile, separate Vercel serverless entry, its own DB read/write URLs). That decoupling is real for the platform API only.
  • 🟡 Net: the consumer-facing web app is a monolith-by-imports; the platform API is a genuine separate service. Two different coupling realities in one repo.

21. Shared Types

  • 🟢 Excellent, codegen-driven type sharing — the strongest area of the codebase.
  • Prisma generates the client + Zod schemas (zod-prisma-types) + Kysely types + an enum generator from one schema.
  • tRPC shares types end-to-end between the web backend and front-end automatically (AppRouter type export).
  • Dedicated @calcom/platform-types, @calcom/platform-enums, @calcom/types, @calcom/tsconfig packages.
  • Zod schemas are embedded directly in the Prisma schema via /// @zod.import(...) annotations, keeping validation and DB shape co-located.
  • 🟢 Essentially zero manual type duplication between layers. This is best-in-class and a major asset for a maintainer.

Quick Wins

  1. 🔴 Purge live-looking secrets from version control. Blank out CRON_API_KEY in .env.example, remove NEXTAUTH_SECRET=secret/CALENDSO_ENCRYPTION_KEY=secret defaults from the Dockerfile, and add a startup assertion that refuses to boot in NODE_ENV=production if any secret equals a known placeholder. High impact, low effort, closes the worst footgun.
  2. 🟡 Audit & document the fork's true feature set. One-page matrix of "claimed removed vs. actually present" (orgs, SAML, billing, Cal.ai). This single artifact de-risks all downstream scoping and prevents the team from operating on a false mental model.
  3. 🟡 Reconcile dependency version skew (Stripe 9/15, Sentry 9/10, Prisma 5/6, Turbo 2.5/2.7, Yarn engines 3.4.1 in apps/web). Aligning these removes a class of subtle cross-workspace bugs and unblocks future upgrades.
  4. 🟡 Remove the DB-at-build-time requirement (or provision an ephemeral DB in CI). Frees up the build pipeline and removes a fragile coupling that bites every new environment.
  5. 🟡 Bump documented Node floor to 20+ and raise the audit gate to fail on high (not just critical) — cheap hardening given the 80-entry resolutions treadmill already in place.
