Codebase Inheritance Assessment — calcom/cal.com (Cal.diy fork)

Prepared by: DawerOne — Principal Engineering Assessment Project type: Fullstack (Next.js + NestJS monorepo)

Executive Summary

This is a large, mature, production-grade open-source scheduling platform (Cal.com, presented here as the MIT-licensed "Cal.diy" community fork). It is a Yarn 4 / Turborepo monorepo of ~7,700 files spanning a Next.js 16 web app, a NestJS-based "API v2" platform, an extensive tRPC backend, a Prisma/PostgreSQL data layer with a very large schema, and a sprawling app-store integration system. The engineering maturity is genuinely high — sophisticated CI security gating, multi-arch Docker releases, changesets-based versioning, distributed tracing, rate limiting, and comprehensive E2E/integration test suites. However, the inherited surface area is enormous and the cognitive load is severe: the Prisma schema alone exceeds 100KB, the codebase carries deep multi-tenant/organization/billing complexity, dozens of third-party OAuth integrations, and numerous deprecated-but-still-present columns and patterns. This is not a project a single developer can "own" quickly. The biggest risks are (1) sheer scale and coupling, (2) a fork that has stripped commercial/EE features but left billing/Stripe/org plumbing throughout, (3) secret-management hygiene with committed default keys, and (4) dependency bleeding edge (Next 16, React 18 pinned but React 19 resolutions present). Treat any "first safe deploy" as a multi-week effort even for an experienced team.

1. Project Summary

  • Purpose: Self-hostable scheduling/booking platform (Calendly alternative). Users define event types, manage availability, connect calendars/video apps, and accept bookings.
  • Current state: Highly mature, actively maintained upstream. Web app is at version 6.2.0. This repo is the Cal.diy fork — README claims EE features (Teams, Organizations, Insights, Workflows, SSO/SAML) were "removed," but the codebase contradicts this: the Prisma schema, tRPC routers, and env config still contain extensive Organization, Team, SAML (@boxyhq/saml-jackson), billing, and platform-OAuth machinery. The "removal" is partial/cosmetic at the schema and infrastructure level.
  • Maturity: Production-grade tooling and processes. The challenge is not immaturity — it's scale and the half-completed fork story.

Rating: 🟡 Mature but the fork's "what's removed" narrative is unreliable; verify feature boundaries empirically.

3. Architecture

A multi-app, multi-package monorepo:

  • apps/web — Next.js 16 app (App Router + legacy pages). The primary product. Hosts the tRPC API endpoint, NextAuth, embeds, and SSR booking pages.
  • apps/api/v2 — NestJS "Platform API" (@calcom/api-v2, license UNLICENSED) for programmatic/OAuth-client access. Deployable to Vercel serverless via a singleton Express adapter (main.ts) or standalone. Uses Bull/Redis throttling, Passport JWT, Swagger.
  • packages/trpc — The real backend brain: a viewer router with ~30 sub-routers (bookings, eventTypes, availability, slots, admin, webhook, apps, oAuth, etc.).
  • packages/prisma — Single source-of-truth schema, generates Prisma client, Zod types (zod-prisma-types), and Kysely types.
  • packages/features — Domain logic (auth, credentials, profiles, users) organized in repository/service patterns.
  • packages/app-store — Massive plugin system for third-party integrations, with a code-gen CLI (app-store-cli).
  • *packages/embeds, packages/platform/, packages/ui, @coss/ui** — Embed libraries, platform atoms (npm-published @calcom/atoms), design system.

Patterns: Repository + Service layering in newer features/* code; tRPC procedure routers; NestJS modules/guards/interceptors in api-v2. There are two parallel API surfaces (tRPC for the web app, REST for api-v2) sharing the same Prisma schema and platform-libraries.

Rating: 🟡 Well-structured but dual-backend complexity and very wide module graph.

4. Dependency Health

  • 🔴 Bleeding-edge Next.js 16.2.3 with React pinned to 18.2.0 in apps/web while resolutions force react@19.2.x → 19.2.4. This React 18/19 ambiguity is a real footgun.
  • 🟡 Version divergence across workspaces: Stripe 9.16.0 (web) vs 15.4.0 (api-v2); Turbo 2.7.1 (root) vs 2.5.5 (web); Sentry 10.33 vs 9.46; Microsoft Graph types 0.15 vs 0.42.
  • 🟡 Huge resolutions block (~80 pinned transitive deps) — this is a maintenance burden and a strong signal of historical CVE firefighting (axios, form-data, jsonwebtoken, qs, tar, lodash, etc. all pinned).
  • 🟡 Patched packages via .yarn/patches: @prisma/client 5.4.2 patch, dayjs, libphonenumber-js, next-i18next. Patches are fragile across upgrades.
  • 🟡 NextAuth v4 (4.24.13) is in maintenance mode; Auth.js v5 migration is a future cost.
  • 🟢 A security-audit.yml workflow and Snyk protect (api-v2) exist — there is active supply-chain attention.
  • 🟢 Engines require Yarn >=4.12; lockfile is committed.

Rating: 🟡 Actively managed but the resolutions/patches sprawl and cross-package version drift are real debt. EOL/maintenance concerns: NextAuth v4.

5. Infra & Deployment

  • 🟢 Extensive GitHub Actions (~50+ workflows): type-check, lint, unit, integration, multiple E2E suites (web, embed, embed-react, app-store, api-v2), bundle analysis, prisma-migration checks, multi-arch Docker release (release-docker.yaml — amd64 + arm64), changesets, and numerous cron workflows (booking reminders, SMS/email/whatsapp reminders, digests, webhook triggers, timezone changes).
  • 🟢 Strong CI security posture: pr.yml uses pull_request_target with a dedicated trust-check gate that blocks untrusted external PRs from running with secrets until a maintainer adds a run-ci label — and explicitly defends against backdated-commit timing attacks. This is mature.
  • 🟢 Docker Compose stack (web, api, postgres, redis, prisma studio) and a Dockerfile-based image published to DockerHub.
  • 🟡 Build-time DB dependency: README notes a database is required at build time (DOCKER_BUILDKIT=0 workaround needed). This complicates CI/CD and reproducible builds.
  • 🟡 Cron jobs run as GitHub Actions workflows hitting endpoints with CRON_API_KEY — operationally fragile for self-hosters who fork without GH Actions, and couples scheduling to GitHub.
  • 🟡 Multiple host targets documented (Vercel Pro required, Railway, Render, Northflank, Elestio) — flexible but each path has caveats.

Rating: 🟢 CI/CD is a genuine strength; 🟡 build-time DB coupling and GH-Actions-as-cron are operational landmines.

6. Code Health

  • 🟢 Biome for lint/format across the repo; lint, type-check, type-check:ci turbo tasks; tsc-absolute matcher.
  • 🟢 Repository/Service abstractions in features/* (e.g., UserRepository, CredentialRepository, ProfileRepository) show deliberate architecture.
  • 🟡 Mixed router/page paradigms in web (App Router + legacy pages, both .ts and .tsx routers).
  • 🟡 next-auth-options.ts is a ~45KB single file with deeply nested sign-in/jwt callbacks containing legacy-fix branches ("FIX LEGACY ISSUE WHERE identityProviderId was accidentally set to userId"). High-risk, high-complexity, security-critical code that is hard to reason about.
  • 🟡 Numerous // deprecated columns left in schema (EventType.price/currency, Booking.scheduledJobs, weightAdjustment, organizationId) — dead-ish data model surface.
  • 🟢 Typing coverage is strong (TS strict-ish, Zod-generated types, Kysely types). eslint-disable @typescript-eslint/no-explicit-any appears (createContext) but sparingly.
  • 🟢 Substantial automated test coverage (unit + integration + many E2E projects).

Rating: 🟡 Healthy tooling, but hotspot files (auth) and deprecated-but-present surfaces raise the onboarding cost.

7. Completeness

  • 🟢 Core scheduling, booking, availability, calendars, event types, webhooks, embeds, app-store integrations, 2FA, OAuth login (Google/Azure), SAML — all present and functional upstream.
  • 🟡 Fork inconsistency: README says Organizations/Teams/Workflows/SSO/Insights were removed, yet schema + env + routers retain Team, Organization, OrganizationSettings, OrganizationOnboarding, SAML, billing (Stripe org/team price IDs), platform OAuth, credit balances, Cal.ai voice agents, and Retell AI. The "community edition" boundary is not clean — expect partially-wired or dead features.
  • 🟡 Billing/credits/proration models (CreditBalance, MonthlyProration, SeatChangeLog) are present — likely non-functional in a license-free fork but still in the data model.
  • 🟢 deploymentSetup / setup-wizard flow exists for first-run.

Rating: 🟡 Feature-complete as a platform, but the fork's claimed scope vs. actual code is misleading — completeness must be verified per-feature.

8. Landmines

  • 🔴 Committed default secrets / weak placeholders. .env.example ships CRON_API_KEY='0cc0e6c35519bba620c9360cfe3e68d0' (a real-looking value, not a placeholder), and docs repeatedly warn that NEXTAUTH_SECRET/CALENDSO_ENCRYPTION_KEY default to literal secret. Self-hosters who skip key generation get a trivially compromised instance. The cron key being a concrete string invites copy-paste into production.
  • 🔴 Encryption key correctness is load-bearing and silently catastrophic. Auth code decrypts 2FA secrets/backup codes with CALENDSO_ENCRYPTION_KEY and only validates secret.length !== 32 at runtime. A misconfigured key bricks 2FA logins.
  • 🟡 NODE_TLS_REJECT_UNAUTHORIZED=0 is recommended in README for SSL-terminating load balancers — globally disabling TLS verification is dangerous guidance.
  • 🟡 Auth signIn callback complexity — multiple email-rematch, account-merge, and "prevent account pre-hijacking" branches. This is exactly where authn bugs (account takeover) live; it has clearly accumulated incident-driven patches.
  • 🟡 allowDangerousEmailAccountLinking: true on Google and Azure providers — intentional but a known account-linking risk class; relies entirely on the email-verified/xms_edov checks being correct.
  • 🟡 Build requires a live database — surprising coupling that will bite CI and reproducible builds.
  • 🟡 Dual Stripe versions and broad billing plumbing in a "no-commercial" fork → latent runtime errors if billing paths are accidentally hit.
  • 🟡 app-store integrations are partially code-generated (*.generated.ts); regeneration drift can silently break integrations.

Rating: 🔴 Secret hygiene and encryption-key fragility are the top must-fix items.

9. Production Readiness Gaps

  1. 🔴 Rotate/strip the concrete CRON_API_KEY from .env.example; force generation of NEXTAUTH_SECRET and CALENDSO_ENCRYPTION_KEY (fail fast on default secret).
  2. 🔴 Add a startup assertion that rejects boot if encryption/auth secrets are missing, default, or wrong-length — rather than failing at first 2FA login.
  3. 🔴 Resolve the React 18-pinned vs. React 19-resolutions ambiguity before any deploy; confirm a single React major actually runs.
  4. 🟡 Decide the fork's true feature boundary: either fully remove Org/Team/billing/SAML or fully support them. Half-wired billing in production is a liability.
  5. 🟡 Remove build-time DB dependency or document a deterministic build DB; otherwise CI is brittle.
  6. 🟡 Replace GitHub-Actions-cron with a self-host-friendly scheduler for any non-GitHub deployment.
  7. 🟡 Pin and unify cross-workspace versions (Stripe, Sentry, Turbo) to avoid behavioral drift.
  8. 🟡 Establish a load/error-budget baseline for the slots/availability hot paths (heavy event-type router, slot caching env vars suggest known load issues).
  9. 🟢 Confirm Sentry, rate limiting (Unkey/throttler), and tracing are configured per environment (infra exists; needs wiring verification).

Rating: 🔴 Not safe to deploy as-is without secret/key hardening and the React version resolution.

11. Data Model

  • Scale: PostgreSQL via Prisma; schema file >100KB — one of the largest single schemas you'll inherit. Dozens of models: User, UserPassword (split out to avoid hash leakage — good), Team, Profile, Membership, EventType, Booking, Attendee, Credential, DestinationCalendar, SelectedCalendar, Host/HostGroup/HostLocation, OrganizationSettings, CreditBalance + credit logs, MonthlyProration, SeatChangeLog, Cal.ai/Agent/phone, BookingReport, WrongAssignmentReport, etc.
  • Relationships: Highly interconnected, multi-tenant (User↔Profile↔Team/Organization with self-referencing org hierarchy via Team.parentId). Round-robin host weighting, managed event types (parentId), and OOO redirects add graph complexity.
  • Indexing: 🟢 Generally thorough — Booking has many composite indexes ([startTime,endTime,status], [userId,status,startTime], etc.) showing query-pattern awareness.
  • Soft-delete: 🟡 Inconsistent — BookingReference.deleted and Attendee.noShow flags exist, but most deletes are hard onDelete: Cascade. No global soft-delete convention.
  • Deprecated columns: 🟡 Several retained (EventType.price/currency, Booking.scheduledJobs, Host.weightAdjustment, User.organizationId) — comments say "plan to drop." Migration debt.
  • Migrations: packages/prisma/migrations exists; check-prisma-migrations CI guard present; auto-migrations.ts build step. 🟢

Rating: 🟡 Well-indexed but enormous; cascade-heavy deletes and deprecated columns warrant care.

12. API Surface

  • Two surfaces:
  • tRPC (packages/trpc) — primary web backend; viewer router → ~30 sub-routers. Type-safe, superjson transformer, custom errorFormatter. No REST versioning; it's RPC.
  • REST "API v2" (apps/api/v2, NestJS) — versioned (/api/v2, also /v2), Swagger/OpenAPI documented (@nestjs/swagger, generate-swagger), with an oasdiff ignore file (~476KB) indicating active OpenAPI diff governance. 🟢
  • Versioning: v2 is explicit; there's evidence of a prior v1 (CreationSource enum API_V1/API_V2). 🟢
  • Docs: Swagger generation pipeline exists for api-v2; tRPC is self-documenting via types. 🟢
  • 🟡 Maintaining parity between tRPC and REST over the same domain is ongoing cost.

Rating: 🟢 Both surfaces are documented/versioned; 🟡 dual maintenance burden.

13. Auth & Authorization

  • Authentication: NextAuth v4, JWT session strategy. Providers: Cal credentials (email/password + TOTP/backup codes), Google, Azure AD, Email magic links, SAML (BoxyHQ). API v2 uses Passport JWT + OAuth clients + API keys (API_KEY_PREFIX=cal_).
  • Authorization: Role-based — UserPermissionRole (USER/ADMIN), MembershipRole (MEMBER/ADMIN/OWNER), plus a newer PBAC/custom Role model (seed-pbac script, customRoleId on Membership). Org-scoped permissions via Profile/Membership.
  • Notable strengths: 🟢 Password split into UserPassword; rate limiting on login (checkRateLimitAndThrowError keyed on hashed email); "INACTIVE_ADMIN" downgrade forcing 2FA+strong-password for CAL-identity admins; explicit account-pre-hijacking guard.
  • Known gaps/risks: 🔴/🟡
  • allowDangerousEmailAccountLinking: true on OAuth providers.
  • Massive, patch-laden signIn/jwt callbacks — the single highest-risk area for authz bugs and account-takeover regressions.
  • 2FA/backup-code security depends on a correctly-sized CALENDSO_ENCRYPTION_KEY with no boot-time validation.
  • E2E bypass: NEXT_PUBLIC_IS_E2E skips admin password/2FA requirements — must be guaranteed unset in prod.

Rating: 🟡 Sophisticated and defense-aware, but concentrated complexity and key-dependency are real liabilities.

14. Async Layer

  • Trigger.dev — primary async tasker (dev:trigger, deploy:trigger:prod, gated by ENABLE_ASYNC_TASKER). Webhook/email taskers behind flags (TASKER_ENABLE_WEBHOOKS, TASKER_ENABLE_EMAILS).
  • BullMQ + Redis in api-v2 (@nestjs/bull) for queues.
  • Cron — implemented as GitHub Actions workflows (booking reminders, SMS/email/WhatsApp reminders, monthly digest, webhook triggers, timezone changes, downgrade users, app-meta sync), authenticated by CRON_API_KEY. Also a cron npm dep and cron-tester.ts in web.
  • 🟡 Cron-via-GitHub-Actions is non-portable for non-GitHub self-hosting and couples scheduling to repo infra.
  • 🟡 Multiple async mechanisms (Trigger.dev, Bull, GH cron, node-cron) — fragmented; understand which is authoritative per deployment.

Rating: 🟡 Capable but fragmented; scheduling portability is a concern.

15. Environment Variables

  • 🟡 ~200+ env vars. turbo.json globalEnv enumerates the full surface; .env.example (17.6KB) + .env.appStore.example (4.1KB) document them. dotenv-checker validates .env against the schema (env-check:common, env-check:app-store).
  • 🟢 Example files and a checker exist — better than most.
  • 🔴 Secret hygiene issues: concrete CRON_API_KEY value committed; secret defaults for crypto keys; multiple encryption keys (CALENDSO_ENCRYPTION_KEY, CALCOM_APP_CREDENTIAL_ENCRYPTION_KEY, CALCOM_SERVICE_ACCOUNT_ENCRYPTION_KEY, CAL_VIDEO_RECORDING_TOKEN_SECRET, JWT_SECRET) each load-bearing and easy to misconfigure.
  • 🟡 No first-class secrets manager in OSS path (Infisical referenced via i-dev/i-dx scripts but optional). Production secret management is left to the operator.
  • 🟡 Build-time vs run-time variable split (documented in README) adds operational complexity for Docker.

Rating: 🔴 The volume + committed-key + default-secret combination is the dominant config risk.

16. State Management

  • Server state: TanStack Query 5.17 (via tRPC React).
  • Client/global state: Jotai 2.12 (atomic) as primary; nuqs 2.8 for URL-synced state; kbar for command palette state.
  • 🟢 Modern, reasonable choices; minimal global mutable state — most state is server-derived via tRPC/Query.
  • 🟡 Mix of Jotai + URL state + React Hook Form across a huge UI; consistency across the app should be verified.

Rating: 🟢 Sound approach; complexity is inherent to app size, not the state strategy.

17. Routing

  • Web: Next.js 16 App Router (primary) with residual legacy pages; --turbopack dev. Dynamic routes for booking pages (username/event-type), org subdomains (multi-tenant routing via ALLOWED_HOSTNAMES, RESERVED_SUBDOMAINS, single-org mode NEXT_PUBLIC_SINGLE_ORG_SLUG). Middleware referenced (couldn't read middleware.ts — confirm CSP/host rewriting there).
  • API v2: NestJS routing with explicit RedirectsMiddleware/RewriterMiddleware, dual /api/v2 and /v2 prefixes.
  • 🟡 Subdomain-based org routing + host allowlists is operationally subtle (the README's CLIENT_FETCH_ERROR troubleshooting underscores this).
  • 🟢 Embeds have dedicated routing/build pipeline.

Rating: 🟡 Powerful multi-tenant routing with real DNS/host configuration complexity.

18. Design System

  • Component libraries: @calcom/ui and @coss/ui (two UI packages — possible migration in progress 🟡), plus @calcom/atoms (npm-published platform component library) built via changesets.
  • Primitives: Radix UI (avatar, dialog, dropdown, popover, tooltip, etc.), react-select, sonner toasts.
  • CSS: Tailwind CSS 4.x with @tailwindcss/postcss. classnames for composition. Theming via next-themes + brand color fields on User/Team.
  • 🟡 Two UI packages (@calcom/ui + @coss/ui) suggests an in-flight design-system migration — verify which is canonical.
  • 🟢 Tokenized theming (brand/dark colors per user/team) and a publishable atoms library indicate design-system maturity.

Rating: 🟡 Mature but possibly mid-migration; confirm the UI package strategy.

19. Performance Flags

  • 🟢 Bundle budget enforced: nextBundleAnalysis budget 358KB with CI annotation (nextjs-bundle-analysis.yml).
  • 🟡 Availability/slots is a known hot path — a dedicated heavy eventTypesHeavy router, slot caching env vars (SLOTS_CACHE_TTL, ENABLE_SLOTS_WORKERS, SLOTS_WORKER_POOL_SIZE), and reservation-tuning vars (NEXT_PUBLIC_QUERY_RESERVATION_*, QUICK_AVAILABILITY_ROLLOUT) all signal that availability computation is a performance pressure point under load.
  • 🟡 MAX_OLD_SPACE_SIZE 4096 build-time and README advice to set --max-old-space-size=16384 for dev indicate heavy memory footprint in build/dev.
  • 🟢 SSR/SSG mix; image handling via sharp/@vercel/og; embeds lazy-loaded.
  • 🟡 Performance-tests workflow exists but availability under concurrency is the area to stress before scaling.

Rating: 🟡 Strong tooling; availability/slots and build memory are the watch items.

20. Coupling Assessment

  • Frontend ↔ tRPC backend: Tightly coupled — the web app is the tRPC backend (same Next.js process). They cannot be deployed independently; tRPC types are imported directly.
  • api-v2 (NestJS): Loosely coupled to the web app and independently deployable (separate Dockerfile, separate Vercel handler, own throttling/Redis). It shares the Prisma schema and platform-libraries packages, so it's coupled at the data/domain layer, not the runtime layer.
  • 🟡 Both backends depend on the same @calcom/prisma schema and many @calcom/platform-* packages — a schema change ripples across both. True service independence is limited to runtime, not domain.

Rating: 🟡 Web+tRPC inseparable; api-v2 deployable independently but domain-coupled via shared packages/schema.

21. Shared Types

  • 🟢 Strong, code-generation-based type sharing. Prisma generates the client; zod-prisma-types generates Zod schemas; prisma-kysely generates Kysely types — all from one schema. tRPC shares end-to-end types from server to React client automatically. Platform packages (@calcom/platform-types, @calcom/platform-enums, @calcom/platform-constants) provide shared contracts consumed by both web and api-v2.
  • 🟢 No manual duplication of core domain types; the monorepo workspace makes @calcom/* imports first-class.
  • 🟡 The api-v2 REST DTOs (class-validator/class-transformer) are a separate type system from tRPC's Zod-based one — the two API surfaces don't share request/response types, only the underlying Prisma/platform types.

Rating: 🟢 Best-in-class internal type sharing; 🟡 minor seam between REST DTOs and tRPC schemas.

Quick Wins

  1. 🔴 Purge committed secrets from examples & fail-fast on defaults. Remove the concrete CRON_API_KEY value from .env.example; add a boot-time guard that refuses to start if NEXTAUTH_SECRET/CALENDSO_ENCRYPTION_KEY equal secret or are the wrong length. Eliminates the single most likely production compromise vector with minimal effort.
  2. 🔴 Resolve the React 18 vs 19 ambiguity. Lock to one React major across apps/web deps and root resolutions, then smoke-test. Prevents subtle hydration/runtime bugs on Next 16.
  3. 🟡 Unify cross-workspace dependency versions (Stripe 9 vs 15, Sentry 9 vs 10, Turbo 2.5 vs 2.7) and fix the false engines.yarn: 3.4.1 in apps/web. Removes a class of "works in one package, breaks in another" surprises.
  4. 🟡 Document the real fork boundary. Produce a one-page matrix of which Org/Team/SAML/billing/Cal.ai features are actually supported vs. dead-but-present in the schema. Saves every future dev days of confusion and prevents accidental use of non-functional billing paths.
  5. 🟡 Make builds DB-optional (or pin a throwaway build DB) and decouple cron from GitHub Actions for self-host targets. Two changes that materially improve deployability and reproducibility.
