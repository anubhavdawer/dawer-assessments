DawerOne Codebase Onboarding Assessment — calcom/cal.com (rebranded "Cal.diy")

Executive Summary

This is the Cal.com scheduling platform monorepo, presented in this repository as a community fork rebranded "Cal.diy" (the README explicitly markets it as the MIT-only, enterprise-stripped edition, though large amounts of enterprise-oriented schema and code remain). It is a large, mature, production-grade Turborepo monorepo (~7,690 files) built on Next.js 16, React 18, tRPC, Prisma/PostgreSQL, and a separate NestJS "API v2" platform layer. The engineering quality is high by open-source standards: strong CI/CD with a security-gated PR pipeline, Biome linting, full TypeScript, Playwright E2E, changesets, and Dockerized deploys. However, the codebase is enormous, deeply coupled to Cal.com-internal services, and carries significant operational complexity — 150+ environment variables, ~50 app-store integrations, two parallel API generations (legacy + NestJS), and a Prisma schema exceeding 100KB with dozens of entities. The single largest risk for any party inheriting this is sheer surface area and the gap between the "self-host MIT" marketing and the reality of a system designed around Cal.com's hosted infrastructure (Stripe billing, Vercel, Trigger.dev, Unkey, Axiom, Sentry, Retell AI, etc.). This is not a small ownership commitment.

1. Project Summary

  • Purpose: Scheduling/booking infrastructure (Calendly competitor): event types, availability, bookings, calendar integrations, video, workflows, teams/organizations, payments.
  • Maturity: Very high. @calcom/web is at v6.2.0; this is a battle-tested product with years of history, extensive migrations, and a mature CI pipeline.
  • Current state: Active, healthy commit cadence. The repo is a fork/rebrand ("Cal.diy") that claims to strip enterprise features — but the Prisma schema still contains organizations, delegation credentials, platform OAuth clients, SAML (@boxyhq/saml-jackson), billing, credits, and Cal.ai voice agents. The "enterprise removed" claim in the README is only partially true and should be verified, not trusted.
  • Rating: 🟢 Mature product / 🟡 fork provenance and "what's actually removed" needs auditing.

3. Architecture

Monorepo structure (Turborepo + Yarn workspaces):

  • apps/web — the main Next.js application (web app + legacy /api Next routes + tRPC host).
  • apps/api/v2 — a standalone NestJS "Platform API" (separate runtime, Bull/Redis, Swagger, JWT/OAuth2, deployed serverless on Vercel via a singleton Express handler).
  • apps/website, apps/ai, apps/console, apps/api-proxy, apps/swagger — supporting apps.
  • packages/features/* — the bulk of domain logic (auth, bookings, webhooks, tasker, credentials, profiles, users).
  • packages/app-store/* — ~50+ third-party integrations (Google, Office365, Zoom, Stripe, HubSpot, Salesforce, etc.).
  • packages/prisma — schema, migrations, Zod/Kysely codegen.
  • packages/trpc — the typed API router tree.
  • packages/ui, @coss/ui — design system.
  • packages/platform/, packages/embeds/ — embeddable React "Atoms" + platform SDK.

Patterns: Repository pattern (UserRepository, CredentialRepository, ProfileRepository), service classes, tRPC router-per-domain, NestJS modules per domain. Reasonably clean layering.

Rating: 🟢 Architecture is coherent and modular for its size. 🟡 The two-API-generation split (Next /api legacy + tRPC + NestJS api-v2) is a significant cognitive and operational tax — three ways to expose backend functionality.

4. Dependency Health

  • Runtime: Node 20 (Dockerfile) — 🟢 supported. README still claims Node ">=18.x" — 🟡 inconsistent; Node 18 is EOL (April 2025).
  • Resolutions block: The root package.json carries an extraordinarily large resolutions map (~80 pins) — many are clearly security-driven overrides (axios 1.15, form-data 4.0.4, jsonwebtoken 9.0.0, tar 7.5.11, node-forge 1.4.0, qs 6.14.2). This signals the team is actively patching transitive CVEs, which is good — but the volume indicates a large, hard-to-audit transitive dependency surface.
  • Patched packages: 5 yarn patches including the 29KB Prisma client patch and dayjs patches. 🟡 Maintenance burden on upgrade.
  • Version skew: React 18 declared vs React 19 resolutions (see §2). Multiple Stripe SDK versions (9.x web vs 15.x api-v2). @microsoft/microsoft-graph-types-beta differs between web (0.15) and api-v2 (0.42).
  • Audit posture: CI runs yarn npm audit --severity critical and fails on critical — 🟢 a real gate, though only critical-level.

Rating: 🟡 Actively maintained but heavy. The override count and ORM patch are debt hot-spots.

5. Infra & Deployment

  • CI/CD: Extensive — 60+ workflow files. The pr.yml pipeline is sophisticated, with a security-gated trust-check that prevents pull_request_target secret exposure to untrusted forks (requires run-ci label). 🟢 This is best-practice supply-chain hygiene.
  • Pipeline coverage: type-check, lint, unit, api-v2 unit, security audit, prisma migration check, integration, E2E (web/api-v2/app-store/embed/embed-react), bundle analysis. 🟢 Comprehensive.
  • Cron: ~12 GitHub-Actions cron workflows (booking reminders, SMS/WhatsApp/email reminders, digest emails, webhook triggers, timezone changes, downgrades). 🟡 Scheduling logic lives in GitHub Actions cron, which is a non-obvious operational dependency — a self-hoster on Docker-only will silently lose these.
  • Docker: Multi-stage Dockerfile with turbo prune. 🟡 Requires a live DATABASE_URL at build time (documented), which is awkward. ARM requires separate image tags.
  • Hosting: Vercel-first (api-v2 ships a Vercel serverless handler; org subdomain provisioning calls Vercel + Cloudflare APIs). Railway/Render/Northflank/Elestio templates provided.
  • IaC: None — no Terraform/Pulumi. Deployment is platform-template-driven.

Rating: 🟢 CI/CD maturity is a genuine strength. 🟡 Deep Vercel/GitHub-Actions coupling undermines the "self-host anywhere" pitch.

6. Code Health

  • Linting/formatting: Biome 2.3.10 across the monorepo; pre-commit via Husky + lint-staged. 🟢
  • Typing: Full TypeScript with tsc-absolute CI type-check; type-check requires 12GB heap (--max-old-space-size=12288). 🟡 The 12GB heap requirement for type-checking is a red flag for codebase size/inference cost — onboarding devs on modest machines will struggle.
  • Tests: Vitest (unit), Playwright (E2E), Jest (api-v2). Coverage exists but is broad rather than guaranteed-deep; E2E is gated behind labels.
  • Naming/consistency: Generally consistent (repository/service patterns, router-per-domain). The auth file contains substantial inline comments documenting legacy-bug workarounds ("FIX LEGACY ISSUE WHERE identityProviderId was accidentally set to userId") — 🟡 evidence of accreted complexity in the most security-sensitive code.
  • Dead code / TODOs: Visible // TODO: apply ratelimits in endpoints.module.ts — rate limiting is stubbed at the NestJS endpoints module level (though a CustomThrottlerGuard exists globally).

Rating: 🟡 High discipline, but the auth and booking subsystems carry years of compatibility cruft.

7. Completeness

  • Done & production-proven: Bookings, event types, availability, calendar integrations, video, teams/orgs, SAML, payments, webhooks, embeds, platform API. This is a complete product.
  • Partial/feature-flagged: Cal.ai voice agents (Retell), credits/billing, organizations self-serve, quick-availability rollout (NEXT_PUBLIC_QUICK_AVAILABILITY_ROLLOUT=10 — percentage rollout), single-org mode.
  • Stubbed: Endpoint-level rate limiting (TODO), CSP "strict" mode (README: "strict isn't supported yet").
  • "Removed" but not really: Despite the fork's claim that orgs/SSO/workflows/insights are stripped, the schema and dependencies (@boxyhq/saml-jackson, org models, platform OAuth) are present. 🔴 The fork's marketing does not match the code — a buyer must not assume enterprise code is absent.

Rating: 🟡 Feature-complete product, but provenance/feature-removal claims require verification.

8. Landmines

  • 🔴 Insecure default secrets in Docker: NEXTAUTH_SECRET=secret and CALENDSO_ENCRYPTION_KEY=secret are baked as Dockerfile ARG defaults. The README warns to override them, but the defaults will silently produce an insecure instance if a self-hoster skips the step. CALENDSO_ENCRYPTION_KEY protects 2FA secrets and backup codes (AES-256) — a default value is catastrophic.
  • 🔴 Committed CRON_API_KEY: .env.example ships CRON_API_KEY='0cc0e6c35519bba620c9360cfe3e68d0' — a real-looking value in an example file invites copy-paste into production. Cron endpoints would be exposed.
  • 🟡 Build-time DB requirement: Docker build needs a live database, complicating CI/CD and air-gapped builds.
  • 🟡 150+ global env vars in turbo.json — configuration is sprawling and error-prone; many cross-cutting flags (NEXT_PUBLIC_*) bake into the client bundle at build time, so changing them requires rebuild (the start script does runtime find/replace as a workaround — itself a landmine).
  • 🟡 Auth callback complexity: next-auth-options.ts (~45KB) contains intricate account-merging, identity-provider linking, and legacy-bug fixes. Account-linking logic is historically the #1 source of auth vulnerabilities (pre-hijacking). There are mitigations present, but this file is a high-risk change surface.
  • 🟡 allowDangerousEmailAccountLinking: true on Google and Azure AD providers — deliberate, with verification guards, but inherently sensitive.
  • 🟡 NODE_TLS_REJECT_UNAUTHORIZED=0 suggested in README for load-balancer SSL termination — disables TLS verification globally if misapplied.
  • 🟡 Type-check requires 12GB RAM — a practical landmine for contributor onboarding and CI cost.

9. Production Readiness Gaps

  1. 🔴 Rotate/remove the committed CRON_API_KEY from .env.example; never ship live-looking secrets.
  2. 🔴 Remove insecure secret defaults for NEXTAUTH_SECRET / CALENDSO_ENCRYPTION_KEY from the Dockerfile, or hard-fail at boot if unset in production.
  3. 🟡 Resolve React 18-vs-19 version skew before any production build is trusted on Next 16.
  4. 🟡 Establish the cron strategy for non-Vercel/non-GitHub deployments — the reminder/webhook crons are GitHub-Actions-driven and won't run in a plain Docker deploy.
  5. 🟡 Implement the stubbed endpoint rate limiting in api-v2 (endpoints.module.ts TODO) and confirm UNKEY/throttler coverage on all public endpoints.
  6. 🟡 Provision Redis (required by api-v2 Bull + throttler) — not mentioned in the basic self-host path.
  7. 🟡 Confirm what enterprise code was actually removed vs. left dormant in the schema; dormant org/billing tables on an MIT "personal" deploy are confusing and risky.
  8. 🟡 Define observability defaults (Sentry/Axiom/PostHog are all optional but wired throughout).

11. Data Model

  • ORM: Prisma (PostgreSQL), schema >100KB — one of the largest single Prisma schemas you'll encounter. Codegen feeds Zod (zod-prisma-types), Kysely (prisma-kysely), and an enum generator.
  • Core entities: User, UserPassword (split out to avoid leaking hashes — 🟢 good), Profile, Team (doubles as Organization via isOrganization/parentId), Membership, EventType, Booking, Attendee, Credential, DestinationCalendar, Host/HostGroup/HostLocation, Schedule/Availability, Webhook, CreditBalance/Credit logs, CalAiPhoneNumber/Agent.
  • Relationships: Highly interconnected; EventType alone has ~40 relations/fields. Team is heavily overloaded (team + org + platform billing + managed orgs).
  • Indexing: 🟢 Thoughtful — composite indexes on Booking ([startTime,endTime,status], [userId,status,startTime], [eventTypeId,status]) show query-pattern awareness.
  • Soft-delete: Inconsistent — BookingReference.deleted flag exists, but most models hard-delete via onDelete: Cascade. No global soft-delete convention. 🟡
  • Migration state: Mature migrations dir; CI has a dedicated check-prisma-migrations job and oasdiff API-diff (476KB ignore file) — 🟢 disciplined migration governance.
  • Tech debt in schema: Multiple deprecated columns explicitly annotated (price, currency, weightAdjustment, scheduledJobs, organizationId on User) — 🟡 deprecations linger, increasing read-path ambiguity.

Rating: 🟢 Well-indexed and codegen-driven; 🟡 overloaded Team model and lingering deprecated columns.

12. API Surface

  • Three surfaces: (1) tRPC — viewer router with ~30 sub-routers (bookings, eventTypes, availability, slots, admin, apiKeys, webhooks, etc.); type-safe, internal-facing. (2) Legacy Next.js /api routes (integration callbacks, webhooks, stripe). (3) NestJS API v2 — public Platform REST API with Swagger/OpenAPI, JWT/OAuth2 auth, versioned under /api/v2 and /v2.
  • Versioning: Explicit /v2 plus a CreationSource enum (API_V1, API_V2, WEBAPP) tracked on records — 🟢 mature.
  • Documentation: Swagger generated for api-v2; oasdiff enforces no-breaking-changes in CI — 🟢.

Rating: 🟢 Strong for the public platform API. 🟡 Three overlapping surfaces is a maintenance and consistency burden.

13. Auth & Authorization

  • Authentication: NextAuth (JWT strategy) with Credentials (bcrypt + TOTP 2FA + backup codes), Google OAuth, Azure AD, SAML (BoxyHQ), and Email magic links. api-v2 uses JWT + OAuth2 (Passport).
  • 2FA: TOTP with AES-256-encrypted secrets and encrypted backup codes — 🟢 solid implementation.
  • Authorization: UserPermissionRole (USER/ADMIN), MembershipRole (MEMBER/ADMIN/OWNER), plus a custom Role model (RBAC). Admin actions gated; "INACTIVE_ADMIN" downgrade if an admin lacks 2FA/strong password — 🟢 thoughtful.
  • Rate limiting: Login path calls checkRateLimitAndThrowError keyed on hashed email — 🟢. But api-v2 endpoint-level rate limiting is a TODO — 🟡.
  • Known gaps/risks: allowDangerousEmailAccountLinking: true (mitigated); extensive legacy account-merge logic; hasValidLicense = false hardcoded in session callback (a deliberate fork artifact, but worth noting). E2E mode bypasses admin password/2FA checks (NEXT_PUBLIC_IS_E2E) — ensure this is never truthy in prod.

Rating: 🟡 Strong primitives, but the merge/linking logic is complex and security-critical.

14. Async Layer

  • Internal Tasker: A typed abstraction (packages/features/tasker) with internal and redis backends; typed task payloads (sendWebhook, sendSms, no-show webhooks, CRM events, analytics, email, booking audit, webhook delivery).
  • Bull + Redis: api-v2 uses @nestjs/bull with Redis (TLS in prod).
  • Trigger.dev: Optional async tasker (ENABLE_ASYNC_TASKER) with prod/staging deploy scripts.
  • Cron: ~12 GitHub-Actions scheduled workflows drive time-based jobs (reminders, digests, webhook triggers).
  • 🟡 Fragmentation: Four async mechanisms (internal tasker, Bull, Trigger.dev, GitHub cron) with overlapping responsibilities. A self-hoster must understand which are active in their deployment to avoid silently dropped reminders/webhooks.

Rating: 🟡 Capable but fragmented; high operational literacy required.

15. Environment Variables

  • .env.example is 17.6KB; turbo.json lists 150+ globalEnv entries. A separate .env.appStore.example (4.1KB) exists for integrations.
  • Secrets management: Infisical is referenced in scripts (infisical run, i-dev, env-gen). No secrets are committed except the problematic CRON_API_KEY value and the secret Dockerfile defaults (see §8).
  • Validation: dotenv-checker env-check tasks run before dev (env-check:common, env-check:app-store) — 🟢.
  • 🔴 Remove the live-looking CRON_API_KEY from .env.example.

Rating: 🟡 Well-documented but overwhelming; two committed-default landmines.

16. State Management

  • Server state: TanStack React Query 5.17 (via tRPC) — the dominant data layer.
  • Client/global state: Jotai (atomic) 2.12 + nuqs (URL state) 2.8. No Redux. kbar for command palette.
  • Approach: Server-state-first with minimal global client state — 🟢 a modern, low-complexity choice appropriate for the app.

Rating: 🟢 Sound.

17. Routing

  • Next.js 16 App Router (with legacy /api Pages routes for webhooks/integrations). --turbopack in dev.
  • Dynamic routes: Username/org-subdomain booking pages, embeds, app-store callbacks. Org subdomains require real DNS (Vercel/Cloudflare provisioning) — a notable infra coupling.
  • Code splitting: Heavy use of dynamic import() in auth and elsewhere; bundle budget enforced (358KB, nextBundleAnalysis) with CI analysis — 🟢.

Rating: 🟢 Modern routing with bundle governance; 🟡 org subdomain routing has infra dependencies.

18. Design System

  • Two UI packages: @calcom/ui (Radix-based primitives, Lexical rich-text editor, react-select, react-day-picker, class-variance-authority) and @coss/ui. 🟡 Two design-system packages is a consolidation smell.
  • Granular exports: @calcom/ui uses explicit per-component export maps (good for tree-shaking, sideEffects: false).
  • Styling: Tailwind CSS 4.x + tailwind-merge. Consistent token-driven theming (brandColor/darkBrandColor on User/Team).
  • Embeddable "Atoms": @calcom/atoms — published npm React components for the platform.

Rating: 🟢 Mature, tree-shakeable; 🟡 dual UI packages.

19. Performance Flags

  • 🟢 Bundle budget enforced in CI (358KB) with PR annotations; granular UI exports; sideEffects: false.
  • 🟡 React 18 on Next 16 prevents leveraging Next 16/React 19 perf features and risks hydration issues until skew is resolved.
  • 🟡 sharp, jsdom, handlebars, full googleapis, @boxyhq/saml-jackson, multiple Stripe SDKs — heavy server dependencies inflate cold-start, especially in the api-v2 serverless path (NestJS bootstrap on Vercel is cold-start-sensitive; the singleton handler mitigates warm invocations).
  • 🟡 Slots/availability is a known hot path — env flags for caching (SLOTS_CACHE_TTL, worker pools, quick-availability rollout) indicate ongoing perf tuning under load.
  • 🟢 Performance-tests CI workflow exists.

Rating: 🟡 Actively managed, but serverless NestJS cold starts and version skew are real risks at scale.

20. Coupling Assessment

  • Frontend ↔ Backend: The web app co-hosts tRPC and legacy /api routes — frontend and the legacy backend are not independently deployable; they ship as one Next.js app.
  • api-v2 (NestJS): Genuinely decoupled — separate runtime, separate deploy, communicates over HTTP/OpenAPI. The platform "Atoms" SDK consumes it. 🟢
  • Shared core: Both depend on packages/features, packages/prisma, packages/platform-* — so they're code-coupled at the monorepo level even when runtime-decoupled.
  • External coupling: Heavy ties to Vercel (serverless handler, domain provisioning), Cloudflare, Trigger.dev, Unkey, Stripe, Sentry/Axiom — the "self-host" story is more aspirational than turnkey.

Rating: 🟡 api-v2 is cleanly separable; the web app + legacy API are monolithic; everything is bound by shared packages and third-party platform assumptions.

21. Shared Types

  • 🟢 Best-in-class type sharing. tRPC gives end-to-end inferred types between the web app and its backend (AppRouter exported). Prisma codegen produces shared Zod schemas (zod-prisma-types), Kysely types, and enums consumed across packages. The api-v2 shares @calcom/platform-types, @calcom/platform-enums, @calcom/platform-constants with the web app and the published SDK. No manual duplication of core types; class-validator/class-transformer add DTO validation on the NestJS side.

Rating: 🟢 A genuine strength.

Quick Wins (Outsized Impact)

  1. 🔴 Purge committed secrets/defaults: Remove the live-looking CRON_API_KEY from .env.example and the secret defaults for NEXTAUTH_SECRET/CALENDSO_ENCRYPTION_KEY from the Dockerfile; make production boot hard-fail when they're unset. One-day change, removes the two worst landmines.
  2. 🟡 Resolve the React 18 vs 19 / Next 16 version skew and document the supported matrix. This de-risks every build and unlocks framework perf features.
  3. 🟡 Write a "true self-host" runbook that lists what's actually required (Redis, cron strategy outside GitHub Actions, DB-at-build, which async backend is active) and what the fork removed vs. left dormant. Closes the gap between the README's promise and operational reality.
  4. 🟡 Implement the stubbed api-v2 endpoint rate limiting (endpoints.module.ts TODO) and audit that every public endpoint is covered by the throttler. Small surface, direct security/abuse impact.
  5. 🟡 Consolidate the UI packages (@calcom/ui + @coss/ui) and pick a single async/job mechanism as canonical — both reduce long-term cognitive load for any team taking ownership.
